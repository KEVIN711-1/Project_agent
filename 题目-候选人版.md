# 智能合约编码题目：Trustless Agent System

## 题目背景

请参考 `ERC-8004` 的部分设计思路，实现一个最小可用的链上 Agent 信任系统（Trustless Agent System）。

系统需要支持以下核心能力：

- Agent 身份注册与管理
- 用户发布任务并锁定奖励
- Agent 提交任务结果
- 用户通过签名确认任务完成
- 系统为 Agent 累积信誉分
- 同时支持 EOA 签名和合约钱包签名校验

本题重点不在于复现完整协议，而在于完成一个结构清晰、逻辑正确、安全边界合理的最小实现。

---

## 实现要求

你可以将系统拆分为 2 到 4 个合约，也可以自行决定模块边界，但至少需要满足以下要求。

### 1. Agent Identity Registry

实现一个 `ERC721` 合约，每个 NFT 表示一个 Agent 身份。

每个 Agent 至少包含以下字段：

- `owner`
- `agentId`
- `metadataURI`
- `active`

可选扩展字段：

- `validator`
- `endpointHash`

要求：

- 只有 Agent owner 可以更新 `metadataURI`
- 支持停用与恢复 Agent
- 需要设计合理事件
- 需要处理不存在的 Agent、越权更新、无效输入等情况

### 2. Task Agreement / Settlement

实现任务创建与结算流程。

每个任务至少包含以下字段：

- `taskId`
- `agentId`
- `client`
- `rewardToken` 或原生币
- `rewardAmount`
- `deadline`
- `resultHash`

要求：

- 用户创建任务时需锁定奖励
- Agent 提交 `resultHash`
- 用户通过 `EIP-712` 签名确认任务完成
- 合约校验签名后向 Agent 支付奖励
- 同一确认签名不能被重放
- 超时后允许用户退款

### 3. Reputation Registry

任务完成后，为对应 Agent 更新信誉数据。

至少记录以下信息：

- 累计完成任务数
- 累计评分
- 最近一次更新时间

要求：

- 只能由结算模块更新信誉
- 允许用户在完成任务时提交 `0-100` 的评分
- 需要防止重复记分

### 4. EIP-1271 支持

系统必须同时支持两类签名校验：

- 如果 `client` 是 EOA，使用 `ECDSA` 校验签名
- 如果 `client` 是合约钱包，使用 `EIP-1271` 校验签名

## 交付物

请提交以下内容：

- Solidity 合约代码
- 单元测试
- 一份简短设计说明

设计说明至少需要说明：

- 任务状态机如何设计
    1、None（默认值）： 未使用该 taskId
    2、Funded（createTask）： 客户已 createTask 并锁赏金
    目标 Agent 必须已注册且处于营业状态
    3、ResultSubmitted（submitResult）：Agent 已 submitResult，链上有 resultHash 
    submitResult 仅允许从 Funded 进入 ResultSubmitted，且须在 deadline 内，且 msg.sender 须为该任务 agentId 的 NFT owner。
    4、Completed（completeTask）：客户签名验收通过，已打款并记信誉 
    completeTask 仅允许在 ResultSubmitted 且 未过 deadline 时进入 Completed。
    5、Refunded （refund）： 超时后客户退款，赏金已退回 
    refund 仅允许在 deadline 已过，且状态为 Funded 或 ResultSubmitted（未完成结算）时进入 Refunded，且须 msg.sender == client。
- 如何处理 nonce 或防重放
    TaskEscrow.sol 为每个 client 维护 mapping(address => uint256) nonces（TaskEscrow）。

    防止旧签名重放流程：
        客户签名时，digest 绑定 当前 nonce 以及 taskId、agentId、resultHash、client、score 等。
    completeTask 验签通过后执行 nonces[client]++。
        之后同一客户再签任意新任务，digest 中的 nonce 已变；旧签名对应的 digest 与链上重算结果不一致，同一 signature 无法再次通过验签。    
    每个任务都有独立的任务id和任务结果hash，签名只能签一个任务是不能签其他的任务
    
    防止跨任务签名：
        struct 含 taskId（任务id）、resultHash（任务结果hash），不能拿 A 任务签名去结 B 任务。
    防止跨链 / 错合约：
        EIP-712 domain 含 chainId、verifyingContract（由 OpenZeppelin EIP712 处理）。
- 如何避免重复结算、重复评分、签名重放

    1、避免重复结算：
        同一任务二次结算：成功后状态为 Completed，completeTask 首行状态检查即失败（只有resultsubmitted 才能进入），与 nonce 共同构成双保险。
    2、避免重复评分：
        客户签完名 → 前端（或 relayer）调 completeTask → 验签通过 → 状态变为 Completed → 之后 completeTask 要求状态仍是 ResultSubmitted，会直接失败，同一条任务不能再次走结算，也就不会再次 recordCompletion。

        ReputationRegistry 里的 taskScored[taskId]：即使将来有人改合约、多路径调用 recordCompletion，同一 taskId 也会在信誉合约层被挡掉。
        一个taskid 评分完后会标记状态为true ，无法第二次评分
    3、避免签名重放
        nonces[client]：防的是签名重放，和「同一任务重复评分」是不同维度，让「旧签名不能再结一次账」。
